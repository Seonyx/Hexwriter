# Claude Code Brief: BookML Remediation Skill for Openclaw (Phase 3, Track B)

## Context

You are building an Openclaw skill that processes BookML work order manifests — XML files that list specific prose remediation tasks (replace overused words, fix repetitive phrases, resolve proximity echoes) targeting paragraphs identified by PID in a BookML chapter XML file.

The work order manifests are generated by a separate analysis engine. The analysis engine also provides a CLI validator (`content-analyser`) that is already deployed on this machine as a self-contained Linux binary. The skill's job is to read the manifest, send targeted rewrite prompts to an LLM, validate the results using the CLI, and produce a revised BookML package ready for re-import to the cloud-hosted editor.

Read `phase3-dispatcher-spec.md` for the full design. This brief scopes your work to the Openclaw skill implementation.

## What to Build

An Openclaw skill at `~/.openclaw/workspace/skills/bookml-remediation/` that processes work order manifests in a state-machine-driven, resumable workflow.

### 1. SKILL.md

The skill definition file. This tells the Openclaw agent what the skill does and how to execute it. The key requirements for SKILL.md:

**This is a batch-processing skill, not a conversational skill.** The agent must not treat status messages as completed turns. The SKILL.md must instruct the agent to:
- Suppress all conversational output until the run completes.
- Execute the numbered gate sequence to completion (or until the per-run entry limit is reached).
- Only emit a summary message after the final checkpoint is written.

**Trigger phrase:** The skill activates when the agent receives a message containing "Process BookML work order manifest" followed by parameters.

**Parameters parsed from the message:**
- `Input:` — path to the input folder containing the BookML package and manifest
- `Output:` — path to the output folder for the revised package
- `Manifest:` — filename of the work order manifest XML within the input folder
- `Model:` — model identifier for LLM calls (e.g., `openai/gpt-5.2`)
- `MaxEntriesPerRun:` — maximum LLM calls per invocation (default: 5)
- `InterCallDelay:` — seconds to wait between LLM calls (default: 30)

**The SKILL.md must define the gate sequence explicitly** as a numbered checklist that the agent follows mechanically. Do not rely on the agent inferring the workflow from a narrative description — spell out each gate as a concrete instruction with filesystem verification at each step.

### 2. State Manager

**File:** `lib/state-manager.ts` (or `.js` — match the existing skill conventions on this machine)

Reads and writes `state.xml` in the output folder.

**Initialise:** Creates state.xml with job metadata (manifest name, chapter ID, source/target draft, model, pacing parameters, started-at timestamp) and all seven gates set to `pending`.

**Resume:** Reads existing state.xml, finds the first gate not marked `completed`, and within that gate finds the first entry not marked `completed`/`failed`/`skipped`. Returns the resume point.

**Update entry:** Sets an entry's status (`completed`, `failed`, `skipped`) with timestamp and optional diagnostic note.

**Update gate:** Sets a gate's status when all its entries are resolved.

**Checkpoint exit:** When the per-run entry limit is reached, writes current state and returns a signal to the main loop to stop processing.

**Finalise:** Sets `completed-at`, writes the summary element (total/completed/failed/skipped/elapsed), sets job status to `completed`.

State.xml namespace: `https://bookml.org/ns/workorder/1.0`

State.xml structure — see spec §5.1 for the full example.

### 3. Manifest Parser

**File:** `lib/parse-manifest.ts`

Parses the work order manifest XML. Returns a structured list of entries with: id, type, severity, status, subject, description, targets (list of PIDs with optional count), constraints (bans, limits, instruction text).

Namespace: `https://bookml.org/ns/workorder/1.0`

Also reads the manifest's `source-draft` attribute (needed for draft increment logic) and `chapter-id`.

### 4. Prompt Builder

**File:** `lib/build-prompt.ts`

Constructs the LLM prompt for a given work order entry. Three templates, one per entry type:

**Outlier word template:**
```
You are editing a fiction manuscript. Rewrite the following paragraph(s) to eliminate or reduce usage of the word "{word}".

Constraints:
- Do NOT use the word "{word}" in your rewrite.
{for each ban: - Do NOT use the word/phrase "{ban.term}".}
- Preserve the paragraph's meaning, tone, and narrative flow.
- Preserve all character names and proper nouns.
- Return ONLY the rewritten paragraph text for each PID, prefixed with [PID: xxx].
- Maintain approximately the same word count (within 20%).

{for each target PID:}
[PID: {pid}]
{paragraph text from chapter XML}

```

**Repetitive n-gram template:**
```
You are editing a fiction manuscript. Rewrite the following paragraph(s) to eliminate the repeated phrase "{phrase}".

Constraints:
- Do NOT use the phrase "{phrase}" in your rewrite.
{for each ban: - Do NOT use the phrase "{ban.term}".}
- Replace with a varied alternative appropriate to the character and scene.
- No two replacement paragraphs should use the same alternative.
- Preserve the paragraph's meaning, tone, and narrative flow.
- Return ONLY the rewritten paragraph text for each PID, prefixed with [PID: xxx].
- Maintain approximately the same word count (within 20%).

{for each target PID:}
[PID: {pid}]
{paragraph text from chapter XML}

```

**Proximity echo template:**
```
You are editing a fiction manuscript. In the second paragraph below, rephrase to avoid repeating the word "{term}" which appears in the preceding paragraph.

Constraints:
- Do NOT use the word "{term}" in the second paragraph.
- Only rewrite the second paragraph. Return the first paragraph unchanged.
- Preserve the paragraph's meaning, tone, and narrative flow.
- Return ONLY the paragraph text for each PID, prefixed with [PID: xxx].

[PID: {firstPid} — DO NOT MODIFY]
{first paragraph text}

[PID: {secondPid} — REWRITE THIS]
{second paragraph text}
```

The entry's `<instruction>` text from the manifest is appended to the end of the prompt if present.

The prompt builder also needs to read paragraph text from the working chapter XML by PID. It takes the chapter XML path and the list of target PIDs, extracts the `<para>` elements by matching `pid` attributes, and returns their text content.

**BookML namespace for chapter parsing:** `https://bookml.org/ns/1.0`

### 5. Response Parser

**File:** `lib/parse-response.ts`

Parses the LLM response to extract rewritten paragraphs. The expected format is:

```
[PID: CH01-P-7K2Q9ZPL]
Rewritten paragraph text here...

[PID: CH01-P-A3BM8NRW]
Another rewritten paragraph here...
```

The parser extracts PID-to-text mappings. If the response doesn't follow this format (LLM hallucinated extra content, included explanations, etc.), the parser should attempt best-effort extraction — look for `[PID: ...]` markers and take the text between them. If no PIDs can be extracted, return a parse failure that triggers validation failure for the entry.

### 6. Validator

**File:** `lib/validate-rewrite.ts`

After applying a rewrite to the working chapter XML, invokes the content-analyser CLI as a subprocess:

```bash
/path/to/content-analyser validate \
  --original {inputFolder}/{chapterFile} \
  --rewritten {outputFolder}/{chapterFile} \
  --entry-id {entryId} \
  --manifest {inputFolder}/{manifestFile}
```

The path to the `content-analyser` binary should be configurable — either hardcoded to a known deployment path (e.g., `/home/steve/tools/content-analyser/content-analyser`) or read from an environment variable (`CONTENT_ANALYSER_PATH`).

Check exit code: 0 = pass, non-zero = fail. Capture stderr for diagnostic messages.

Return a result object: `{ passed: boolean, diagnostics: string[] }`.

### 7. Chapter Updater

**File:** `lib/update-chapter.ts`

Applies validated rewrites to the working chapter XML.

**Apply rewrite:** Takes the PID-to-text mappings from the response parser, finds the matching `<para>` elements in the working chapter XML by `pid` attribute, and replaces their text content. Writes the updated XML back to the working copy in the output folder.

**Snapshot and revert:** Before applying a rewrite, saves the current text of affected paragraphs. If validation fails, restores them. This ensures a failed rewrite doesn't corrupt the working copy.

**BookML namespace:** `https://bookml.org/ns/1.0`

### 8. Draft Incrementer

**File:** `lib/draft-incrementer.ts`

After all entries are processed (gate 6 in the state machine), builds the complete output BookML package:

1. Copy `book.xml` from input folder to output folder. Modify it:
   - Find the existing draft entry with `status="in-progress"` or `status="snapshot"` — set it to `snapshot` if not already.
   - Add a new `<draft>` element: `number="{targetDraft}" status="in-progress" based-on="{sourceDraft}" created-at="{timestamp}" created-by="openclaw-remediation"`.
2. Update the working chapter XML: set `draft="{targetDraft}"` on the root `<chapter>` element and all `<component>` entries.
3. Copy all other files from input to output unchanged (metadata XMLs, notes XMLs, other chapter folders that aren't in the manifest).

The target draft number is `sourceDraft + 1`, read from the manifest's `source-draft` attribute.

### 9. Main Orchestrator

The main loop that SKILL.md instructs the agent to execute. This can be described procedurally in SKILL.md itself (the agent follows the instructions) or implemented as a script the agent runs. Match whichever pattern existing Openclaw skills use for complex workflows.

**The gate sequence:**

1. **Initialise:** Check for state.xml in output folder. If resuming, load state and jump to the right gate. If fresh, parse manifest, copy chapter XML to output as working copy, write initial state.xml.

2. **Process outlier words:** Loop through `outlier-word` entries in severity order. For each: check run counter against `MaxEntriesPerRun` (exit cleanly if reached), wait `InterCallDelay` seconds if not the first call, build prompt, call LLM, parse response, apply to working copy, validate via CLI, update state. Failed validations revert the working copy and mark the entry failed.

3. **Process n-grams:** Same loop for `repetitive-ngram` entries. The per-run counter carries over from gate 2.

4. **Revalidate echoes:** Run `content-analyser analyse` against the working chapter XML. For each `proximity-echo` entry still pending, check whether the echo term still appears within the echo window in the updated text. Mark resolved echoes as `skipped`. No LLM calls in this gate.

5. **Process echoes:** Same loop for surviving `proximity-echo` entries.

6. **Build output package:** Call draft-incrementer to assemble the final BookML package.

7. **Finalise:** Write completion summary to state.xml. Emit completion message to the agent conversation.

## Verification

1. **Fresh run:** Point the skill at a test chapter with a work order manifest. Verify it creates state.xml, processes entries up to `MaxEntriesPerRun`, checkpoints, and exits.
2. **Resume:** Run the skill again with the same input/output folders. Verify it reads state.xml and resumes from where it stopped — no re-processing of completed entries.
3. **Completion:** After enough invocations to process all entries, verify:
   - state.xml shows `status="completed"` with correct summary counts.
   - The output folder contains a complete BookML package with incremented draft number.
   - The revised chapter XML has the expected changes (banned words removed from completed entries).
4. **Validation failure:** Manually corrupt a rewrite (or use a model that produces bad output) and verify the skill marks the entry `failed`, reverts the paragraph in the working copy, and continues to the next entry.
5. **Echo revalidation:** After processing outlier words and n-grams, verify that gate 4 correctly identifies echoes that no longer exist and marks them `skipped`.
6. **Inter-call delay:** Observe that the skill pauses between LLM calls for the configured delay.
7. **Import test:** Take the output package, zip it, upload to the cloud editor, and import. Verify it's recognised as the correct next draft number.
